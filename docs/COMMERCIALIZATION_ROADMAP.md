# LILIA Commercialization Roadmap

この文書は、LILIA を小規模商用化するための検証順、MVP境界、価格仮説、やらないことを管理する正本である。
開発ロードマップとMVP境界は `docs/ROADMAP.md` を正本にし、この文書は商用化・販売・有料ベータ・画像生成付きプレイの検証順だけを扱う。

## 1. Purpose

LILIAを小規模商売として始めるための検証順、MVP境界、価格仮説、やらないことを管理する。

最初から大規模AI彼女アプリや総合プラットフォームを目指さない。
少人数の有料ベータで、記憶・関係・人格変化・物語品質に対して継続課金する熱量があるかを見る。

## 2. Commercial Position

LILIAはマス向けAI彼女アプリではなく、記憶・関係・人格変化・物語品質を重視するプレミアムニッチ向け。

差別化の中心は、画像枚数、音声、Live2Dの豪華さではない。
ユーザーの言葉と選択が残り、resume時に声・距離・余韻が戻り、1人のLILIAとの関係が少しずつ育つことを価値にする。

## 3. Current Hypothesis

WebUI + 1ヒロイン + 記憶継続 + ときどき場面絵、が最初の商用MVP。

CLIの完成度を直接売るのではなく、外部クライアントまたはWebUIから使える形へ包む。
画像生成は常時ではなく、初登場、場面転換、関係節目のような高価値タイミングに絞る。

## 4. Phase 0: 商用化前の通過条件

- new → first scene → 10 turns → save → resume が自然に通る。
- 画像なしでも30分遊べる。
- resume 1ターン目で声・距離・余韻が戻る。
- 1人のLILIAとの会話が、最低限「また続けたい」と思える。
- Save Mode と Play Mode が混ざらず、ユーザー体験にメタログが漏れない。

## 5. Phase 1: 小規模有料ベータ

- 20〜50人。
- 月額980〜1,980円。
- 1ヒロイン。
- 画像は初登場・場面転換・関係節目のみ。
- 継続率と熱量を見る。

見るもの:

- 初回作成からfirst scene完了まで到達するか。
- 2回目ログイン、7日後継続があるか。
- 画像なしターンでも会話継続するか。
- 不満がUIなのか、生成品質なのか、キャラ一貫性なのか。

## 6. Phase 2: WebUI正式版

- 月額1,980〜2,980円。
- 画像上限つき。
- 必要なら追加画像クレジット。
- 成人向け導線は慎重に分離。

この段階で API 化、ユーザーごとのセッション管理、API キー認証、課金・利用量管理を本格化する。
CRAVE ブランドでの BOOTH / DLsite 販売、またはサブスクリプション提供を検討する。

## 7. Phase 3: 回想・漫画化

関係の節目をアルバム化、短い漫画化。
ただし初期MVPでは作らない。

候補:

- JSONL → Markdown 変換: codex rollout JSONL からプレイヤーと LILIA の会話を抽出し、読みやすい Markdown に変換するツール。
- Markdown → 漫画変換: 会話 Markdown からワンシーン（1 枚見せ / 4 コマ形式）の漫画パネルを生成する。画像生成 API（Gemini Imagen 等）との連携を想定。

## 8. Business Risks

- 導入摩擦。
- 生成品質のブレ。
- キャラ一貫性。
- 画像コスト。
- NSFW規約。
- 決済停止。
- サポート負荷。
- 期待値が「何でもできるAI彼女」へ膨らみ、LILIAの関係育成価値とズレること。

## 9. Do Not Build Before Validation

- 複数ヒロイン。
- Live2D。
- 音声。
- 漫画化。
- 戦闘。
- 経営。
- 世界拡張。
- スマホアプリ化。
- 大規模マーケットプレイス。
- 画像生成の常時自動化。

## 10. Success Metrics

- 初回完了率。
- 2回目ログイン率。
- 7日後継続率。
- 平均ターン数。
- 画像生成頻度。
- 月額継続人数。
- save / resume 後の継続ターン数。
- ユーザーが自発的に「保存して」「続きから」と言う頻度。

## 11. Next Priority Wave Candidates

- 3 本フック運用（戦闘なし版）: メイン事件フック / ヒロイン関係フック / 探索・生活フックの 3 方向を並列管理する。戦闘システムは分離し、イベント駆動のフック管理に絞る。タイプ被り禁止（3 本とも恋愛系にしない等）。
- 品質自動検証: 入力に対する応答品質を自動テストする仕組み。現在は kk が手動でゼロから立ち上げて叩いているが、リグレッション検知や Wave 効果測定を自動化したい。
- JSONL → Markdown 変換: 漫画パイプラインの前提として、まず可読化だけを作る。
- Wave Y-E: document_generator / spine_generator への Example Anchoring Control 導入。

## Reference Notes

Claude による市場調査メモでは、AI companion / AI girlfriend 市場、BOOTH / DLsite 販売、API化、WebUI化、生成イラスト、漫画化が候補として挙げられている。

ただし、市場規模・競合売上・年商予測は未検証の外部仮説として扱う。
LILIAの初期判断では、数値予測ではなく、20〜50人の有料ベータで実際の継続率を見る。
